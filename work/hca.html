Making HCA Healthcare's Sites More Accessible

Overview

I'm currently working at HCA Healthcare (an owner of hundreds of hospital facilities and specialty sites) on a contract project to make their websites more accessible. I'm part of a team in the Web Shared Services department. My job is to perform audits and updat the issues we find so the sites will work well for all users.

The Web Shared Services department currently most of HCA's 14 divisions, and our accessibility project is in charge of working on all of the division, facility, and specialty sites. By the end of this project we'll have improved accessibility on several hundred sites.

We have a team of five "digital content specialists", two project managers, and an accessibility QA specialist.

What We Work On

There are a lot of things we review and update for each site:

  • videos on HCA Healthcare Youtube channels need to have captions (not the auto-generated ones) or a transcript for videos without a person speaking
  • all PDF documents need to have appropriate tags and reading order for screenreader users
  • short, text-based PDFs get turned into HTML pages (because HTML is easier to make screenreader friendly)
  • links are changed to make sure they're clear ("read more" gets updated to "our oncology team", for example), since screenreader users can navigate a page by links
  • each site's branding is reviewed for color contrast
  • any old Flash content is marked for removal
  • HTML gets updated to remove things like iframes

Tools We Use

HCA uses a CMS platform called dotCMS to house the sites. It's got some similarities to other CMS platforms like WordPress, but it's designed for enterprises with a lot of sites.

We use Siteimprove to run audits on the websites so we don't have to manually check each of the hundreds of thousands of pages individually.

Brailleworks updates PDF documents for screenreader accessibility.

Rev.com creates captions and transcripts for Youtube videos.

NVDA a screen reader that we use for testing sites

axe Monitor is an auditing tool that looks for accessibility issues on lots of sites and pages at the same time

Other Work

The accessibility project has sprints for each division, which means some of the time we're waiting on the facilities or the vendors and we're less busy. During those times I've worked on:

  • Find Care, a large scale project to replace outdated "service" pages on each site with more cohesive "specialties" pages
  • Training new members of the accessibility team
  • Helping the ticketing team when their queue gets too long
  • Reading articles, watching webinars, and taking online courses to learn more about accessibility so I can contribute more to our project

What I've Learned

Before I started this role I had some basic knowledge about color contrast, only having one H1 per page, and that people use screenreaders. I've been able to get much more in-depth about how easy it can be to create websites that work for all users.

I've also gotten to experience using a screenreader myself (I have to listen to it at non-superhuman speeds though!) - NVDA is a free screenreader created by blind developers, so it's a great one to try out.

There's always more to learn, so even after my contract ends I'll still be reading articles to keep up with the best way to design sites. The things I've learned here will help me create better digital products in the future.
